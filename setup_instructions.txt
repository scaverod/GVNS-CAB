STEP 2.1: CHOOSING THE RIGHT LICENSE
--------------------------------------------------------------------------------

Selecting the appropriate license is crucial for your research software.
Here are some resources to help you choose:

üîó RECOMMENDED RESOURCES:

1. Choose a License (https://choosealicense.com/)
   - Simple, easy-to-understand guide
   - Compares popular licenses side-by-side
   - Provides license text ready to copy

2. TL;DR Legal (https://www.tldrlegal.com/)
   - Plain English explanations of licenses
   - Shows what you can/cannot do with each license
   - Good for understanding implications

3. Open Source Initiative (https://opensource.org/licenses)
   - Official list of approved open source licenses
   - Detailed information about each license

üìã COMMON LICENSES FOR RESEARCH SOFTWARE:

‚Ä¢ MIT License (Default in this template)
  ‚úì Very permissive and simple
  ‚úì Allows commercial use
  ‚úì Only requires attribution
  ‚úì Most popular for academic software
  ‚Üí Good for: Maximum adoption and flexibility

‚Ä¢ Apache License 2.0
  ‚úì Similar to MIT but with patent protection
  ‚úì Requires stating significant changes
  ‚úì Good legal protection for contributors
  ‚Üí Good for: Projects worried about patent issues

‚Ä¢ GNU GPL v3
  ‚úì Requires derivative works to be open source
  ‚úì "Copyleft" - ensures code remains free
  ‚úì Cannot be used in proprietary software
  ‚Üí Good for: Ensuring modifications stay open

‚Ä¢ BSD 3-Clause License
  ‚úì Similar to MIT
  ‚úì Prohibits using author names for endorsement
  ‚úì Very permissive
  ‚Üí Good for: Academic projects wanting attribution control

‚Ä¢ Creative Commons CC0 (Public Domain)
  ‚úì No restrictions whatsoever
  ‚úì Waives all copyright
  ‚úì Good for datasets and benchmarks
  ‚Üí Good for: Maximum freedom, especially for data

üéì FOR ACADEMIC RESEARCH:

Most academic software uses MIT or Apache 2.0 because they:
- Allow others to build upon your work freely
- Are compatible with commercial use
- Require attribution (important for citations)
- Are well-understood by the community

‚ö†Ô∏è IMPORTANT CONSIDERATIONS:

1. Check your institution's policy on software licensing
2. If you have industry collaborators, consult with them
3. If your work is funded by a grant, check funding requirements
4. Consider whether you want derivative works to remain open source
5. For datasets, consider Creative Commons licenses instead

üìù HOW TO APPLY YOUR CHOSEN LICENSE:

1. Visit https://choosealicense.com/ and select your license
2. Copy the license text
3. Create a file named LICENSE in your repository root
4. Paste the license text
5. Update the year and copyright holder name
6. Update the license badge in README.md if you change from MIT
7. Update CITATION.cff with the correct license identifier

Example for changing the license badge in README.md:
Replace: [![License](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
With: [![License](https://img.shields.io/badge/License-Apache_2.0-blue.svg)](LICENSE)

STEP 8.1: CREATING A GITHUB RELEASE (RECOMMENDED)
--------------------------------------------------------------------------------

Creating releases is important because it:
‚úì Provides a stable, citable version of your code
‚úì Generates a DOI through Zenodo integration
‚úì Allows users to download specific versions easily
‚úì Creates a permanent snapshot of your work
‚úì Shows professional software development practices

üîß HOW TO CREATE A RELEASE:

1. Make sure all your changes are committed and pushed to main:
   git add .
   git commit -m "Prepare for v1.0.0 release"
   git push origin main

2. Go to your repository on GitHub

3. Click on "Releases" in the right sidebar (or go to /releases)

4. Click "Create a new release" or "Draft a new release"

5. Fill in the release information:

   üìå Choose a tag:
   - Click "Choose a tag"
   - Type: v1.0.0 (for your first release)
   - Click "Create new tag: v1.0.0 on publish"
   
   Versioning scheme (Semantic Versioning):
   - v1.0.0 = Initial release with paper publication
   - v1.0.1 = Bug fixes, no new features
   - v1.1.0 = Minor improvements or new features
   - v2.0.0 = Major changes or breaking changes

   üìù Release title:
   Example: "v1.0.0 - Initial Publication Release"
   Or: "v1.0.0 - [Paper Title] Implementation"

   üìÑ Description (use this template):
   ```
   This is the initial release of the code and data accompanying our paper:
   
   **Paper Title:** [Your Paper Title]
   **Authors:** [Author list]
   **Journal/Conference:** [Publication venue]
   **DOI:** https://doi.org/10.XXXX/XXXXXXX
   
   ## What's included
   
   - Source code for all algorithms described in the paper
   - Benchmark instances used in experiments
   - Scripts to reproduce experimental results
   - Pre-compiled JAR file for easy execution
   
   ## Citation
   
   If you use this software, please cite our paper:
   [Copy your BibTeX citation here]
   
   ## Quick Start
   
   ```bash
   java -jar target/code.jar
   ```
   
   See README.md for detailed instructions.
   
   ## Computational Environment
   
   - Java version: [X.X]
   - Tested on: [OS and version]
   - Dependencies: See pom.xml
   
   ## Known Issues
   
   None at this time.
   
   ## Support
   
   For questions or issues, please open an issue on GitHub or contact:
   [your.email@institution.edu]
   ```

6. Attach binary files (optional but recommended):
   - Click "Attach binaries by dropping them here or selecting them"
   - Upload your compiled .jar file
   - Upload any additional data files not in the repo
   - These will be directly downloadable from the release page

7. Choose release options:
   ‚òë Set as the latest release (recommended for your first release)
   ‚òê Set as a pre-release (only if this is beta/experimental)
   ‚òê Create a discussion for this release (optional)

8. Click "Publish release"

üì¶ WHAT HAPPENS AFTER PUBLISHING:

- GitHub creates a permanent snapshot (zip and tar.gz archives)
- Users can download source code for this specific version
- If connected to Zenodo, a DOI is automatically generated
- The release appears in your repository's sidebar
- You can reference this specific version in your paper

üîÑ CREATING SUBSEQUENT RELEASES:

When you fix bugs or add features after publication:

1. Make your changes and commit them
2. Update CHANGELOG.md with what changed
3. Create a new release with a new version number:
   - v1.0.1 for bug fixes
   - v1.1.0 for new features
   - v2.0.0 for major changes

4. In the release description, mention:
   ```
   ## What's Changed
   - Fixed bug in algorithm X (#issue_number)
   - Improved performance of function Y
   - Updated documentation
   
   ## Full Changelog
   See CHANGELOG.md for complete details
   ```

üí° BEST PRACTICES:

- Create a release when your paper is accepted/published
- Create new releases for significant bug fixes
- Always include the compiled .jar in release assets
- Update the README badge to show latest release:
  ![GitHub release](https://img.shields.io/github/v/release/username/repository)
- Tag releases with version numbers that match your citations
- Keep release descriptions clear and informative

‚ö†Ô∏è IMPORTANT NOTES:

- Releases are permanent and cannot be deleted (only hidden)
- Source code snapshots are automatically created by GitHub
- Once a release is published, the tag cannot be changed
- Plan your version number carefully before publishing
- Consider creating a pre-release (v1.0.0-rc1) before the final version
